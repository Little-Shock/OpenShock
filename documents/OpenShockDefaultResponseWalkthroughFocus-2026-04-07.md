## 1. 这轮不再沿用什么旧口径

本轮默认响应 QA 不再沿用下面这些旧解释：

1. `monitoring agent` 接未 `@mention` 消息
2. 未 `@mention` 一套流程，显式 `@mention` 另一套流程
3. `default_monitor_response` 先决定谁能说话
4. 默认响应先是任务化表达，再补聊天

这些口径都只代表旧实现，不再作为新的验收标准。

## 2. 新验收基线的 3 条主语义

### 2.1 chat-first

要验证的是：

1. agent 是否先判断这条消息要不要回应
2. 如果回应，首条回复是否先自然语言接话
3. 是否仍然出现“接任务 / 分派任务 / 创建任务”式首反应

重点观察：

1. room / dm 中的首条 agent 回复文案
2. 回复是否像同事交流，而不是系统编排
3. 普通呼叫、提问、请求帮助消息是否被自然接住

### 2.2 `@mention` 只是增强信号

要验证的是：

1. 显式 `@mention` 是否只是增强目标 agent 的回应倾向
2. 不带 `@mention` 和带 `@mention` 的消息，是否仍共享同一类处理语义
3. 是否还存在“被 `@mention` 就切到另一套任务化流程”的行为

重点观察：

1. 同类消息在有无 `@mention` 下的首条回复风格
2. 被 `@mention` 的 agent 是否更强烈地承担回应责任
3. 是否还能看见旧版 `fallback` / `default_monitor_response` 残留痕迹

### 2.3 task 不是默认响应入口

要验证的是：

1. 普通消息是否不会立刻进入 task 语义
2. task 是否只在目标、边界、执行意图已经足够清楚时才进一步形成
3. 默认响应回复里是否仍然把 task 当作首反应

重点观察：

1. 首条回复是否先完成沟通
2. 后续是否只有在信息足够清楚时，才出现 task 化动作或建议
3. 是否存在“一句普通消息 -> 直接 task 化”的现象

## 3. 浏览器走查建议顺序

### 3.1 基线场景：未 `@mention` 的呼叫消息

建议消息：

1. `有人吗？`
2. `这个谁能帮我看一下？`

验证点：

1. 是否有 agent 回应
2. 首条回复是否 `chat-first`
3. 是否没有先变成任务化表达

### 3.2 负向场景：纯状态播报

建议消息：

1. `我刚把文档同步好了。`
2. `这个已经处理完了。`

验证点：

1. agent 是否可以选择不回
2. 系统是否不会把纯状态播报强行转成默认响应

### 3.3 显式 `@mention` 场景

建议消息：

1. `@agent_shell 你看一下这个问题。`
2. `@agent_guardian 这里你先回应一下。`

验证点：

1. 被 `@mention` 的 agent 是否更倾向回应
2. 首条回复是否仍然 `chat-first`
3. 是否没有切成另一套任务化流程

### 3.4 对话继续推进到 task 语义

建议方式：

1. 先发送普通沟通消息
2. 再逐步补充明确目标、边界、执行要求

验证点：

1. 首条回复先完成沟通
2. 只有在边界足够清楚后，才进一步形成 task 分析或动作

## 4. API / 可观测性关注点

这轮除了浏览器观察，还要重点看 payload 是否已经从旧语义切到新语义。

建议关注：

1. event frame / message metadata 里是否明确包含：
   - 当前 target
   - actor
   - 是否有 `@mention`
   - 被 `@mention` 的 agent 列表
2. 是否还保留旧版 `default_monitor_response` 等命名并主导流程
3. 是否还能看到 `monitoring agent` 被当作产品真相

如果还能看到这些旧语义在主链路上起决定作用，应记为本轮回归失败或至少记为未收干净。

---

## 5. 报告里必须单列的观察项

下一轮 `#19` 报告至少要单列下面 4 项：

1. 首条回复是否 `chat-first`
2. `@mention` 是否只作为增强信号
3. task 是否只作为后续分析结果出现
4. 旧 `monitoring / fallback / default_monitor_response` 语义是否已退出主解释面

---

## 6. 结论标准

这一轮如果要给出“默认响应新语义基本对齐”的结论，至少要满足：

1. 普通呼叫/提问消息的首条回复是自然语言接话
2. 纯状态播报允许不回
3. 显式 `@mention` 只增强信号，不触发另一套流程
4. task 不是默认响应入口
5. 旧 `monitoring / fallback` 语义不再是主解释框架

只要其中任何一条不成立，都不能写成“只是文案问题”，而要进一步判断这是：

1. 真 bug
2. 设计落地不完整
3. 提示词仍带旧语义
4. 工程兜底仍然越权成了产品主语义

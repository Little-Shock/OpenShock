## 1. 目标

这份文档不是测试用例本身，而是定义收尾阶段如何把：

1. 测试执行
2. 问题记录
3. 修复分流
4. 回归验证
5. 结论同步

串成一个稳定闭环，避免现在这个阶段最常见的三类失控：

1. 测了很多，但没有形成可执行修复清单。
2. 修了很多，但没有明确回归范围。
3. 需求在微调，但 bug、产品决策、当前实现限制被混在一起。

## 2. 当前阶段的测试基线

在收尾阶段，所有测试反馈都要先建立在当前已知基线之上，否则容易把旧问题、预期内限制、未来目标混成“新 bug”。

   - `apps/frontend`：协作壳、看板、Inbox、Room 交互

   - Issue 创建
   - Repo 绑定

   - Run 执行
   - Merge 审批与集成
   - Delivery PR 创建
   - Webhook 收尾
5. 当前验证基线：
   - `apps/backend`：`go test ./...`
   - `apps/daemon`：`go test ./...`

因此，收尾阶段的新增测试工作，不应再从“系统能不能跑起来”开始，而应从“主链路是否因新改动退化、边界是否被打穿、微调是否引入新回归”开始。

## 3. 测试闭环总流程

每一轮变更或测试，都按下面 6 步走：

### Step 1. 明确本轮测试对象

先明确本轮是在测哪一类变更：

1. 新修复后的回归确认
2. 某个需求微调的影响验证
3. 主链路 smoke
4. 某个具体 bug 的复现与确认

如果对象不清楚，先补一句：

`本轮目标是什么，预期改动影响哪一层，是否影响 action/run/merge 状态闭环`

不满足这句说明时，不进入测试执行。

### Step 2. 做最小预检

执行测试前，至少确认：

1. 受影响服务能启动
2. 基础验证命令能通过或已有明确失败说明
3. 当前测试 repo、runtime、agent、provider 是什么
4. 是否会受到 in-memory 重启丢失影响

建议最小预检：

1. backend：`go test ./...` 或至少 `go build ./...`
2. daemon：`go test ./...` 或至少 `go build ./...`
3. frontend：`npm run build`
4. 如果改动涉及界面交互，再补 `npm run lint`

### Step 3. 执行用例或复现步骤

执行优先级遵循：

1. 先跑主链路 smoke
2. 再跑本次改动直接影响点
3. 最后补边界和异常流

主链路 smoke 默认最少覆盖：

1. 创建或进入一个 Issue Room
2. 验证 Room 消息、Task Board、Inbox 基本可达
3. 如本轮涉及 repo / run / merge，则至少验证对应链路入口可操作
4. 如本轮涉及 realtime，则至少验证页面刷新或状态推送结果

### Step 4. 记录问题并立即分类

每发现一个问题，不能只记“坏了”，必须立即归类为下面 5 种之一：

1. 环境问题
2. 安装或依赖问题
3. 数据或夹具问题
4. 产品功能问题
5. 可观测性不足

并额外判断它属于下面哪一种：

1. 真 bug
2. 需求微调
3. 当前已知限制
4. 未来目标，不属于当前 MVP 缺陷

如果一个问题无法分类，不进入修复排期，先补澄清。

### Step 5. 分流给修复责任层

每个问题都要落到最可能负责的层，不要直接抛成模糊问题：

1. 前端展示/交互问题
2. 后端状态模型/API 契约问题
3. daemon 执行/回传问题
4. 跨层契约问题
5. 产品预期不清，需要先定规则

分流时至少回答 3 个问题：

1. 改的是展示层，还是对象语义？
2. 会不会影响 action/run/merge 的状态闭环？
3. 修复后必须补哪几个回归点？

### Step 6. 修复后回归并关闭

任何修复在关闭前，至少要完成：

1. 原问题复现路径回归
2. 直接相邻链路回归
3. 一条主链路 smoke

验证结论只用下面 4 个结果：

1. `已修复`
2. `部分修复`
3. `未修复`
4. `非 bug，属于需求/预期调整`

## 4. 收尾阶段的分流规则

这是当前阶段最关键的判断标准。

### 4.1 什么应该直接记为 bug

满足下面任一条件，直接记为 bug：

1. 当前 walkthrough 已通过的主链路能力被新改动打坏
2. action / run / merge / delivery 状态不再闭环
3. 前后端契约漂移，导致页面、看板、Inbox、Room 状态不一致
4. daemon 已成功执行，但结果未正确回流系统
5. 用户在 UI 可见路径中无法完成当前文档已声明可执行的操作

### 4.2 什么更像需求微调

满足下面任一条件，优先按需求微调处理：

1. 业务对象还成立，只是展示文案、入口位置、默认行为要调整
2. 页面信息层级、操作顺序、按钮 gating 需要优化
3. 当前实现可以工作，但体验不符合收尾阶段预期

这类问题记录时不要只写“bug”，要补上：

`建议调整后的目标行为`

### 4.3 什么属于当前已知限制，不应误报

默认不直接记为 bug 的场景：

1. 重启后内存态数据丢失
2. 当前实现没有 Postgres、outbox、完整外部平台集成
3. 依赖未来路线图才成立的能力

这类情况如果影响测试执行，应记录为：

`当前限制 / 不纳入本轮缺陷`

### 4.4 什么必须先澄清再测

满足下面任一条件时，先问清楚：

1. 无法判断这是对象语义变化还是 UI 调整
2. 改动会影响 Issue、Task、Run、MergeAttempt 这些核心对象定义
3. 同一个操作在人类理解和系统状态上存在冲突

## 6. 回归策略

### 6.1 每次修复后的最小回归

所有修复完成后，至少跑下面 3 类回归：

1. 原问题复现路径
2. 与该问题同层的相邻操作
3. 一条主链路 smoke

### 6.2 主链路 smoke 包

建议作为默认 smoke 的最小集合：

1. 打开首页，确认 shell 可用
2. 进入一个 Issue Room
3. 验证 Room 消息区、Task Board、Inbox 至少可访问
4. 如涉及任务流，验证 Task 创建或 Task 状态变更
5. 如涉及执行流，验证 run 创建、状态变化或回流
6. 如涉及交付流，验证 merge/inbox/delivery 对应入口和状态

### 6.3 高风险变更的扩展回归

如果改动触及下面区域，扩展回归不能省：

1. `apps/backend/internal/store`
2. `apps/backend/internal/api`
3. `apps/daemon/internal/*`
4. Room 右侧 issue 面板
5. Task Board 的 quick create、status control、run action
6. Inbox 的审批动作

这类改动除定点回归外，必须再补：

1. 一条完整 Issue -> Task -> Run -> Merge 或 Delivery 相关链路
2. 一次刷新页面或重进页面验证

## 7. 节奏建议

MVP 收尾阶段，不建议把测试组织成“大批量最后一次验收”，而应该组织成短闭环：

1. 开发提交一个微调或修复
2. QA 快速复测
3. 立即给出：
   - 通过
   - 不通过
   - 部分通过
   - 需要澄清
4. 如不通过，明确下一轮只修最关键阻塞

建议节奏：

1. 每个高风险改动单独一轮小回归
2. 每半天至少做一次主链路 smoke
3. 每天末尾更新一次“当前可用基线”和“未关闭问题清单”
